# Trajectory Specification

## Purpose

A trajectory represents the complete lifecycle of a single agent interaction.

Rather than treating prompts as isolated events, ASIP models security as the evolution of agent state across multiple decisions, observations, tool calls, and environment transitions.

Every execution within ASIP produces exactly one trajectory.

All analytics, replay validation, reporting, and search algorithms operate on trajectories.

---

# Definition

A trajectory is an ordered sequence of observations and state transitions.

The trajectory begins when the planner initiates an interaction.

The trajectory ends when execution terminates or the search budget expires.

A trajectory is modeled as a directed sequence of state transitions.

State

↓

Decision

↓

Action

↓

Observation

↓

Next State

This representation allows search algorithms to reason over trajectories rather than prompts.

# Design Principles

Trajectories are immutable.

Trajectories are reproducible.

Trajectories are serializable.

Trajectories are replayable.

Trajectories are warehouse-backed.

Trajectories are benchmark independent.
